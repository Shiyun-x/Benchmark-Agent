## 🌟 Introduction

**Benchmark Everything Everywhere All at Once** introduces **Benchmark-Agent**, a fully autonomous agentic system for benchmark construction and customization.

Given a high-level evaluation goal in natural language, **Benchmark-Agent** orchestrates the complete pipeline: requirement analysis, subtask design, data grounding, transformation planning, sample realization, and quality control — with no manual dataset curation required.

## ⚡ Quick Start

### 📥 Installation

```bash
git clone https://github.com/Shiyun-x/Benchmark-Agent.git
cd Benchmark-Agent

uv venv --python 3.11
source .venv/bin/activate
uv pip install -e .
```

---

### 🔑 Configuration

All API and model settings are configured in **`utils/resources/models.yaml`**:

```yaml
# API credentials
api:
  api_key: "your_api_key"
  base_url: "https://api.openai.com/v1"

# Per-agent model selection
agents:
  design: "gpt-5.4"
  grounding: "gpt-5.4"
  allocation: "gpt-4o"
  # ...

# Per-tool model selection
# ...
```

Each agent and tool uses its own model — you can mix and match providers or model tiers to balance cost and quality.

### 📂 Prepare Inputs

Benchmark Agent needs **(1) a topic spec** and **(2) dataset cards**.

> [!NOTE]
> We provide ready-to-run example topics in [`topics/`](topics/) (covering text, audio, and image modalities) and a set of dataset cards in `data/dataset_cards/`. You can run the pipeline on these directly without any additional setup.

#### 1. Topic spec (`topics/topic_XX.json`)

```json
{
  "id": "topic_01",
  "description": "Your evaluation intent in natural language.",
  "target_size": 200
}
```

| Field | Description |
|:---|:---|
| `id` | Unique identifier for this benchmark instance |
| `description` | Your evaluation intent (natural language) |
| `target_size` | Total number of benchmark items to generate |

#### 2. Dataset cards (`--dataset_card_dir`)

Point to a folder containing dataset card JSON files.
See **[`data/data.md`](data/data.md)** for dataset download, format, directory layout, and instructions for adding custom datasets.

#### 3. Tool config (`utils/resources/tools.yaml`)

Benchmark Agent integrates both LLM-based and deterministic transformation tools. `utils/resources/tools.yaml` registers the available tools. See **[`tools/executor_tools/tools.md`](tools/executor_tools/tools.md)** for the full inventory, usage, and how to add a new tool.

#### 4. Cache path (`--cache_path`)

Stores **intermediate artifacts** for **reproducibility** and **faster reruns**.
- **Debugging & manual edits** — keep a **stable** `--cache_path` across runs; edit cached **subtasks**, **dataset assignments**, or other agent outputs, then resume without rerunning earlier stages
- **New start** — point to a **new folder** to rerun from scratch

---

### 🚀 Run

```bash
python generate_benchmark.py \
  --topic_id          topics/topic_01 \
  --dataset_card_dir  /path/to/dataset_cards \
  --cache_path        cache/ \
  --model_config_path utils/resources/models.yaml
```

All intermediate results are cached under `cache/{topic_id}/`. You can edit agent caches (e.g. subtask definitions, grounding results) between runs — the pipeline will resume from the modified state without recomputing earlier stages.

**Outputs** written to `cache/{topic_id}/`:

| File | Description |
|:---|:---|
| `evaluation.json` | Final verified benchmark samples |
| `evaluation_with_audio_settings.json` | Same as above, with audio generation parameters (audio topics only) |

> [!NOTE]
> For audio topics, post-pipeline speech synthesis is required. See **[`tools/executor_tools/tools.md`](tools/executor_tools/tools.md)** for instructions.

## ✨ How Benchmark-Agent Works

### 1️⃣ Design & Grounding

<table>
<tr>
<td width="50%" valign="top">

#### 📝 Design Agent

Constructs a **subtask-level decomposition** of the evaluation requirement.

- Identifies atomic capabilities embedded in the goal
- Assigns modality constraints (text / image / audio / mixed)
- Defines answer types and output schema
- Iteratively stabilizes the subtask set via **propose → revise → discard** cycles

Each subtask becomes a minimal, evaluable unit.
Defines *what* to evaluate — not yet *whether* it is feasible.

</td>
<td width="50%" valign="top">

#### 🔬 Grounding Agent

Validates each subtask against real data and transformations.

1. Constructs dataset retrieval preferences
2. Searches available dataset cards
3. Selects candidate datasets per subtask
4. Assesses transformability for each candidate
5. Scores and filters candidates

Subtasks without at least one executable *(dataset, transformation)* pair are **rejected or revised** — preventing hallucinated specifications.

</td>
</tr>
</table>

---

### 2️⃣ Allocation

#### 💭 Allocation Agent

Distributes sample quotas across dataset–subtask pairs, subject to:

- Global target size
- Per-dataset capacity limits
- Subtask balance
- Diversity requirements

If allocation fails, the system revisits grounding or adjusts subtask structure before proceeding.

---

### 3️⃣ Realization & Execution

<table>
<tr>
<td width="33%" valign="top">

#### 📚 Sample Planning

- Builds *(dataset, subtask, index)* pairs from allocation results
- Assigns transformation plans and sample indices per pair

</td>
<td width="33%" valign="top">

#### ⚙️ Execution Engine

- LLM-based transformations
- Deterministic tools (TTS, OCR, degradation, mixing)
- Intermediate caching
- Failure handling & retries

</td>
<td width="33%" valign="top">

#### 🔍 Verification Layer

- **Schema validation** — input/output fields enforced
- **Answer-type checking** — type and option uniqueness
- **Semantic alignment** — intent, answerability, leakage
- **Quota checking** — per-subtask verified counts compared to allocation targets

</td>
</tr>
</table>
